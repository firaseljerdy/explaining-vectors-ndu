SECTION 01

Why Should I Care About Vectors?

Okay so before we get into the math, let me just explain why vectors matter. Because honestly, when I first heard "vectors" I thought of those arrow diagrams from physics class that I never really vibed with. But here's the thing—every single moving object in a game or simulation uses vectors. Every. Single. One.

Think about a simple bouncing ball. Without vectors, you'd write something like this:

let x = 100;
let y = 100;
let xspeed = 2.5;
let yspeed = 2;

That's four variables just for position and speed. Now imagine you also need acceleration, wind, friction, a target position... suddenly you're drowning in xWhatever and yWhatever variables. In 3D? Triple everything. It becomes a nightmare to manage.

With vectors, that same thing becomes:

let position = createVector(100, 100);
let velocity = createVector(2.5, 2);

Two variables. Clean. Readable. And the vector object gives us built-in math operations for free. That's the whole pitch. It doesn't give us new superpowers right away—it just organizes everything so that when things do get complex (and they will), we're not fighting our own code.

My Takeaway

Vectors aren't about doing new things. They're about doing the same things cleaner, so when you add more physics later, you don't drown in variables. The payoff really hits in Chapter 2 with forces.

SECTION 02

What Even Is a Vector?

A vector is something that has both magnitude (how big) and direction (which way). That's it. That's the whole definition. Compare that to a scalar, which is just a plain number—like 5, or 3.7, or -42. A scalar tells you "how much" but not "which way."

The way I like to think about it: a vector is an arrow. It starts somewhere and points somewhere else. The length of the arrow is the magnitude. The way it's pointing is the direction.

Mathematically, in 2D, a vector is just a pair of numbers: an x and a y. Those numbers are the "instructions" for the arrow—go this far horizontally, then this far vertically.

Key Distinction

Scalar = a single number (speed, temperature, mass)
Vector = magnitude + direction (velocity, force, acceleration)

Speed is a scalar. Velocity is a vector—because velocity includes direction.

One thing that confused me at first: position is also a vector. I was like "but position is just a point, not a direction." True—but if you think of position as "the path from the origin (0,0) to where I am now," it becomes a vector. The origin is the tail, your current spot is the head. That mental model works.

SECTION 03

Vectors in p5.js

p5.js gives us a built-in p5.Vector class. You don't have to write your own (though it's good to understand what's happening under the hood). You create vectors using createVector():

let position = createVector(100, 100);
let velocity = createVector(1, 3.3);

// Access components with dot notation
circle(position.x, position.y, 48);
Watch Out

You can't use createVector() outside of setup() or draw(). The p5 library hasn't loaded yet at the global level. Declare the variable globally, but assign it inside setup().

Under the hood, a vector is just an object with x, y, and z properties plus a bunch of methods. If we were building it ourselves it'd look like:

class Vector {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }
}

That's the core of it—just a wrapper around two numbers. But the methods it provides (add(), sub(), mult(), mag(), normalize(), etc.) are what make it so powerful. You're not reinventing arithmetic every time; you just call the method.

Important: even with vectors, you'll still sometimes need to pull out individual .x and .y values. Functions like circle() don't accept vector objects—they want separate numbers. That's normal. The point isn't to never use x and y again; it's to organize them together.

SECTION 04

Vector Addition

This is the bread and butter operation. Adding two vectors means combining them—component by component:

w = u + v = (ux + vx, uy + vy)
Add the x's together, add the y's together. That's it.

Visually, there's the head-to-tail rule: place the second vector's tail at the first vector's head. The result (called the resultant) goes from the very start to the very end. It's the "shortcut" arrow.

Why does this matter? Because this is literally how motion works. Every frame of a simulation, you do this:

position.add(velocity);

That one line says: "wherever I am now, move me by my velocity." The position vector gets updated. The ball moves. That's it. That's motion.

And when you have multiple forces—like a player moving right while wind pushes left—you just add all the vectors together. The final resultant is where the object actually ends up. The engine (literally, a game engine) adds them all up for you.

The Click Moment

What really made addition click for me: it's not just a math operation. It's a physical simulation primitive. Every time something moves in a game, vector addition is happening. position += velocity is the most fundamental line in all of game physics.

SECTION 05

Vector Subtraction

Subtraction is just addition with the second vector flipped. Negating a vector means flipping its direction: if v = (3, 4), then -v = (-3, -4). Same length, opposite direction.

w = u - v = (ux - vx, uy - vy)

But here's when subtraction becomes genuinely interesting: when you want to find the direction from one point to another.

Say you have an enemy at position A and the player at position B. How does the enemy know which way to move? Calculate B - A and you get a vector that points from A to B. That's the "look at" direction. It's used everywhere—AI targeting, camera follow, homing missiles, you name it.

let mouse = createVector(mouseX, mouseY);
let center = createVector(width / 2, height / 2);
mouse.sub(center);
// mouse now points FROM center TO cursor

After that subtraction, mouse is no longer a position—it's a direction vector pointing from the center of the canvas to wherever the mouse is. This is a super common pattern.

How I Remember It

Target - Me = Direction to Target. That's the formula. If I'm the enemy, and the player is the target: player.pos - enemy.pos = which way to chase. Always subtract "where I am" from "where I want to go."

SECTION 06

Scalar Multiplication & Division

When we "multiply a vector," we don't mean multiplying two vectors together (that's a different, more advanced thing). We mean multiplying a vector by a scalar—a regular number. This scales the vector: changes its length without changing its direction.

w = u × n = (ux × n, uy × n)
Multiply each component by the scalar. Direction stays the same.

Practical examples:

  • velocity.mult(2) — double the speed
  • velocity.mult(0.5) — half the speed (same as dividing by 2)
  • velocity.mult(-1) — reverse direction completely
  • velocity.mult(0) — stop entirely (zero vector)

Division works exactly the same way but with div(). Multiplying by 0.5 and dividing by 2 produce identical results. Use whichever reads better in context.

let u = createVector(-3, 7);
u.mult(3);
// u is now (-9, 21) — 3 times longer, same direction

let v = createVector(8, -4);
v.div(2);
// v is now (4, -2) — half the length

The key insight: scaling doesn't rotate the vector. It keeps pointing the same way (or flips 180° if the scalar is negative). This is exactly what you want when adjusting speed—you want to go faster or slower in the same direction.

SECTION 07

Vector Magnitude

The magnitude of a vector is its length. If the vector is (3, 4), how long is the arrow? This is where our old friend Pythagoras shows up.

Since a vector's x and y components form a right triangle with the vector itself as the hypotenuse:

||v|| = √(x² + y²)
e.g. magnitude of (3, 4) = √(9 + 16) = √25 = 5

In p5.js you just call v.mag() and it does the Pythagorean theorem for you. The magnitude is always positive, even if the components are negative (because they get squared).

let v = createVector(3, 4);
let m = v.mag();
// m = 5

Why do we care about magnitude? Because it tells us speed (if the vector is velocity), distance (if it's a position difference), or strength (if it's a force). You'll use mag() constantly to check things like "is this object moving too fast?" or "how far apart are these two things?"

SECTION 08

Normalizing Vectors

Normalizing is one of those things that sounds fancy but is actually dead simple: make the vector's length exactly 1, while keeping its direction the same. The result is called a unit vector.

How? Divide each component by the magnitude:

û = v / ||v||
Divide each component by the vector's length. Now its length is 1.

For example, a vector (3, 4) has magnitude 5. Normalized: (3/5, 4/5) = (0.6, 0.8). Check: √(0.36 + 0.64) = √1 = 1. Boom, unit vector.

let v = createVector(3, 4);
v.normalize();
// v is now (0.6, 0.8) — length = 1

But why would I want a length-1 vector? Because it isolates the direction. Once you have pure direction, you can scale it to whatever speed you want by multiplying:

let dir = p5.Vector.sub(target, myPos);
dir.normalize();  // pure direction, length = 1
dir.mult(5);       // now length = 5 (my desired speed)
The Pattern

Normalize → Multiply is THE standard pattern for "move toward something at a fixed speed." You'll see this combo literally everywhere in game dev. It's also available as a single function: setMag(speed).

Small gotcha: what if the vector's magnitude is 0? You can't divide by zero. p5.js handles this gracefully—normalize() checks for zero-length vectors and just does nothing. But it's good to be aware of it.

SECTION 09

Motion with Vectors

Now let's bring it all together. The absolute core of motion simulation is two steps:

  1. Add velocity to position
  2. Draw the object at its new position

That's it. That's what Shiffman calls "Motion 101." And now that we have vectors, we can encapsulate this in a class:

class Mover {
  constructor() {
    this.position = createVector(random(width), random(height));
    this.velocity = createVector(random(-2, 2), random(-2, 2));
  }

  update() {
    this.position.add(this.velocity);
  }

  show() {
    circle(this.position.x, this.position.y, 48);
  }
}

The beauty here: the Mover class holds p5.Vector objects as its data. An object containing objects. That's totally normal in OOP—you'll see this pattern everywhere. Later, a ParticleSystem will contain Particle objects, and each Particle will contain p5.Vector objects. It's turtles (well, vectors) all the way down.

The problem with this basic setup: the ball just moves in a straight line forever. No speeding up, no slowing down, no turning. To get interesting motion, we need one more ingredient.

SECTION 10

Acceleration

Acceleration is the rate of change of velocity. Just like velocity is the rate of change of position. It's a trickle-down chain:

acceleration → velocity → position
Acceleration changes velocity. Velocity changes position.
velocity.add(acceleration);
position.add(velocity);

Two lines. That's the entire physics engine in its most basic form. Here's the rule Shiffman sets for himself (and that I think is really smart): never directly change velocity or position. Instead, only change acceleration, and let it trickle down. This forces you to think about motion in terms of forces, which is how the real world works.

Three approaches to acceleration:

1. Constant Acceleration

Set acceleration to a fixed vector like (-0.001, 0.01). The object gradually speeds up in that direction. These values are tiny because they accumulate ~30 times per second. Use velocity.limit(10) to cap the speed so it doesn't fly off screen.

2. Random Acceleration

Use p5.Vector.random2D() to get a random unit vector each frame, then scale it. This creates a jittery, organic motion—way smoother than a random walker because the velocity has continuity. The randomness affects the acceleration (change in velocity), not the velocity directly.

3. Acceleration Toward a Target

This is where things get really cool. Calculate the direction to the mouse (subtraction!), normalize it, scale it, and set that as the acceleration. The object steers toward the cursor. More on this below.

Important Nuance

Acceleration doesn't just mean "speeding up." It means any change in velocity—speeding up, slowing down, or turning. A car turning a corner at constant speed is still accelerating because the direction of its velocity is changing.

SECTION 11

Static vs. Nonstatic Methods

This tripped me up at first so I want to make it really clear. p5.Vector has two versions of most math methods:

Type Syntax What happens
Nonstatic v.add(u) Changes v in place. u is unchanged.
Static p5.Vector.add(v, u) Returns a new vector. Both v and u are unchanged.

The nonstatic version modifies the original. The static version creates a new result and leaves the originals alone. It's the difference between "add B onto A" vs "what is A + B?"

// NONSTATIC: modifies v
let v = createVector(1, 2);
let u = createVector(3, 4);
v.add(u);
// v is now (4, 6), u is still (3, 4)

// STATIC: creates new vector
let v = createVector(1, 2);
let u = createVector(3, 4);
let w = p5.Vector.add(v, u);
// w is (4, 6), v is still (1, 2), u is still (3, 4)
Common Mistake

Writing let w = v.add(u); and expecting w to be the result. Nope—add() doesn't return anything. It changes v in place and you've accidentally mutated your original vector. Use the static version when you need a fresh result.

This applies to add(), sub(), mult(), and div(). In general: use the nonstatic version when you want to modify an object (like updating position), and the static version when you need to calculate something without side effects (like finding a direction).

SECTION 12

Accelerating Toward a Target

This is where everything comes together. Let's make an object that steers toward the mouse. The algorithm:

  1. Find the direction — Subtract my position from the target: target - me
  2. Normalize — Make it a unit vector (length 1)
  3. Scale — Multiply by desired acceleration strength
  4. Apply — Set it as the object's acceleration
update() {
  let mouse = createVector(mouseX, mouseY);
  let dir = p5.Vector.sub(mouse, this.position); // Step 1
  dir.normalize();                               // Step 2
  dir.mult(0.2);                                  // Step 3
  this.acceleration = dir;                       // Step 4

  this.velocity.add(this.acceleration);
  this.velocity.limit(this.topSpeed);
  this.position.add(this.velocity);
}

See how every concept from this chapter comes together? Subtraction to find direction. Normalization to get a consistent unit. Scalar multiplication to control strength. Addition to apply velocity and acceleration. It's all one pipeline.

Why It Overshoots

If you run this, you'll notice the object orbits around the mouse instead of stopping at it. That's because the object has no concept of "I'm close enough, time to brake." It keeps accelerating at the same rate regardless of distance. Fixing this requires arrival behavior—which we'll get to later. For now, the overshoot is actually useful; it shows the object is doing real physics, not just teleporting.

And honestly? This is already dangerously close to gravitational attraction. The only difference is that real gravity's strength is inversely proportional to distance (closer = stronger). We're using a constant strength here. Chapter 2 covers forces properly, and this pattern is the foundation for everything that comes next.

That's Chapter 1. The whole thing boils down to: use vectors to represent position, velocity, and acceleration. Add them in order. Everything else is just variations on that theme.

The math isn't hard—it's addition, subtraction, multiplication, and one square root. The power comes from combining these simple operations into a system that produces complex, natural-looking motion. And from here on out, everything builds on top of this foundation.
