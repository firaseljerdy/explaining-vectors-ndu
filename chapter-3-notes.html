SECTION 01

Why Angles Matter

In Chapter 2, we learned that forces are vectors that cause acceleration. But what about things that spin? A wheel, a pendulum, a baton twirling in the air—these aren't just moving from A to B, they're rotating. To describe rotation, we need angles. And to work with angles in code, we need radians.

Chapter 3 bridges the gap between linear motion (forces pushing objects in straight lines) and oscillation (things swinging, bouncing, and waving back and forth). Before we can get to pendulums and sine waves, we need to be completely comfortable with how angles work in p5.js.

So if Chapter 2 answered "why is something accelerating?" (forces), Chapter 3 starts by answering "why is something spinning?" (angular forces) and then builds toward periodic, repeating motion.

My Takeaway

Linear motion and angular motion are the same pattern wearing different outfits. Force → Acceleration → Velocity → Position becomes Torque → Angular Acceleration → Angular Velocity → Angle. Once you see it, you can't unsee it.

SECTION 02

What Are Radians?

A radian measures an angle by arc length. Take a circle with radius 1. If you walk along the edge of that circle a distance of exactly 1, the angle you've swept is 1 radian. Walk the entire circumference (2π ≈ 6.28) and you've gone a full circle.

θ (radians) = arc length / radius
For a unit circle (radius = 1), the angle in radians equals the arc length directly.

Compare that to degrees: someone long ago decided to split a circle into 360 equal parts. Why 360? Because it's divisible by lots of numbers (2, 3, 4, 5, 6, 8, 9, 10, 12...). It's convenient for humans, but arbitrary for math.

The Key Equivalences

  • Full circle: 2π rad = 360°
  • Half circle: π rad = 180°
  • Quarter circle: π/2 rad = 90°
  • One radian: ≈ 57.3°
The Conversion Formula

To convert between the two: radians = degrees × (π / 180) and degrees = radians × (180 / π). In p5.js you can also use the built-in helpers radians(90) and degrees(PI).

SECTION 03

Degrees vs Radians

So why not just use degrees? After all, "rotate 90 degrees" is more intuitive than "rotate π/2 radians." The answer is that radians make every formula cleaner.

Consider the derivative of sin(x):

  • In radians: d/dx sin(x) = cos(x). Done.
  • In degrees: d/dx sin(x°) = (π/180) × cos(x°). Extra constant everywhere.

When you're running a simulation at 60 frames per second and computing angular velocities, accelerations, and forces every single frame, that π/180 factor multiplied into every calculation is a disaster for clean code. Radians eliminate it entirely.

The Click Moment

p5.js defaults to radians for rotate(), sin(), cos(), and atan2(). If you ever pass a degree value to rotate() by accident, you'll get wildly wrong results—rotate(90) doesn't rotate 90°, it rotates 90 radians, which is over 14 full spins. This burned me exactly once before I started thinking in radians.

You can switch to degrees with angleMode(DEGREES), but the Nature of Code textbook (and most physics simulations) stick with radians. It's worth training your brain to think in terms of PI.

// p5.js defaults to radians
rotate(PI);            // 180°
rotate(HALF_PI);       // 90°
rotate(TWO_PI);        // 360° (full circle)

// Built-in converters
let r = radians(90);  // → PI/2
let d = degrees(PI);  // → 180

// Or switch mode entirely (not recommended for simulations)
angleMode(DEGREES);   // now rotate(90) means 90°
SECTION 04

Rotation in p5.js: translate + rotate

Here's the pattern that every rotation demo in p5.js uses:

push();                          // save current state
translate(width/2, height/2);   // move origin to center
rotate(angle);                    // rotate around new origin
// draw stuff here...
pop();                           // restore state

The order matters: translate first, then rotate. Why? Because rotate() always spins around the current origin (0, 0). If you don't translate first, your shape rotates around the top-left corner of the canvas, which is almost never what you want.

Order Matters!

translate() then rotate() = spin in place (what you want).
rotate() then translate() = orbit around the top-left corner (probably not what you want).

Think of it like this: translate moves the paper, rotate spins the paper. Move the paper to where you want the center of rotation, then spin it.

The push() / pop() pair saves and restores the transformation state. Without them, your translate and rotate would affect everything drawn afterward. It's like putting your changes in a sandbox.

SECTION 05

Class Demo 1: Mouse-Mapped Rotation

This is the first demo we built in class. The idea: map the mouse X position to a full rotation (0 to 2π) and visualize it by spinning a square with a red "arm" line to make the angle obvious.

function setup() {
  createCanvas(400, 400);
}

function draw() {
  background(220);

  push();
  translate(width / 2, height / 2);
  rectMode(CENTER);

  let angle = map(mouseX, 0, width, 0, TWO_PI);
  rotate(angle);

  fill(120);
  square(0, 0, 100);

  stroke(255, 0, 0);
  strokeWeight(3);
  line(0, 0, 100, 0);
  pop();

  fill(0);
  text("Angle in Radians: " + angle.toFixed(2), 10, 20);
}

Key decisions in this code:

  • map(mouseX, 0, width, 0, TWO_PI) — linearly maps the mouse's horizontal position to a full circle of radians. Mouse at left edge = 0 rad, mouse at right edge = 2π rad.
  • rectMode(CENTER) — draws the square centered on the origin instead of from its top-left corner. Without this, the square would wobble asymmetrically.
  • The red arm line — goes from (0,0) to (100,0), which before rotation points straight right. As we rotate, the line makes the current angle visually obvious.
Why This Demo Works So Well

It takes an abstract concept (radians) and maps it to something you can feel—your mouse position. Move the mouse a quarter of the way across → quarter turn. Half way → half turn. All the way → full revolution. Instant physical intuition for what "2π radians" actually means.

SECTION 06

Angular Motion: The Pipeline

Here's the beautiful thing: angular motion is the same pipeline as linear motion. Compare:

velocity += acceleration   →   angularVelocity += angularAcceleration
position += velocity       →   angle += angularVelocity
Linear motion (left) vs Angular motion (right) — same structure, different names.

The difference? Linear motion uses vectors (x, y components). Angular motion uses scalars (single numbers). An angle is just a number. Angular velocity is just a number. No need for a Vec class—plain arithmetic works.

let angle = 0;
let angleVelocity = 0;
let angleAcceleration = 0.01;

function draw() {
  angleVelocity += angleAcceleration;
  angle += angleVelocity;

  push();
  translate(width / 2, height / 2);
  rotate(angle);
  // draw your rotating object...
  pop();
}

With constant angular acceleration of 0.01, the object starts slow and spins faster and faster—exactly like a ball accelerating in a straight line, but in a circle. Acceleration trickles down to velocity, velocity trickles down to angle.

Scalars vs Vectors

In 2D, angular quantities are scalars because there's only one axis of rotation (perpendicular to the screen). In 3D, angular velocity becomes a vector (pointing along the axis of rotation), but we don't need that complexity yet.

SECTION 07

Class Demo 2: Wind-Driven Baton

The second class demo adds external force (wind from the mouse) and damping to the angular motion pipeline. A baton spins based on where you move your mouse, and gradually slows down thanks to the damping factor.

let angle = 0;
let angleVelocity = 0;
let angleAcceleration = 0;

function setup() {
  createCanvas(400, 400);
}

function draw() {
  background(255);

  let windStrength = map(mouseX, 0, width, -0.05, 0.05);
  let wind = createVector(windStrength, 0);
  angleAcceleration = wind.x;

  // Angular motion update
  angleVelocity += angleAcceleration;
  angleVelocity *= 0.95;   // damping
  angle += angleVelocity;
  angleAcceleration = 0;    // reset

  push();
  translate(width / 2, height / 2);
  rotate(angle);
  stroke(0);
  fill(127);
  line(-60, 0, 60, 0);
  circle(60, 0, 16);
  circle(-60, 0, 16);
  pop();
}

Breaking It Down

  • map(mouseX, 0, width, -0.05, 0.05) — mouse on the left produces negative acceleration (counterclockwise), mouse on the right produces positive (clockwise). Center = zero wind.
  • angleVelocity *= 0.95 — this is damping, the angular equivalent of friction. Every frame, 5% of the angular velocity is removed. Without it, the baton would spin faster and faster forever.
  • angleAcceleration = 0 — same reason we reset acc.mult(0) in the Mover class: acceleration is recalculated from forces each frame. If we don't reset, last frame's wind stacks on top of this frame's wind.
Don't Forget to Reset Acceleration

This is the exact same trap from Chapter 2. Forces (and angular forces) are instantaneous. They're applied fresh every frame. If you skip angleAcceleration = 0, you get a runaway baton that spins to infinity because old accelerations keep accumulating.

Damping vs Friction

In Chapter 2, friction was a force we calculated and applied via applyForce(). Damping is simpler: we just multiply velocity by a number slightly less than 1. The effect is the same (motion gradually dies out), but the mechanism is different. Damping is a shortcut—less physically accurate, but easier to implement and perfectly fine for most simulations.

The Damping Sweet Spot

0.95 means 5% of velocity is lost per frame. Values closer to 1.0 (like 0.99) give a smooth, slow deceleration—good for objects in space. Values closer to 0 (like 0.8) give aggressive damping—good for thick liquid. Try changing this value in the demo to see the difference. 1.0 = no damping (perpetual spin). Above 1.0 = the baton accelerates on its own (unstable!).

SECTION 08

The Bridge: Rotation → Oscillation

Today's class covered the foundation: radians, rotation in p5.js, and the angular motion pipeline. But this is all setup for the real payoff: oscillation.

Here's the key insight that connects everything: if you take a point rotating around a circle and project its position onto a single axis (say, the y-axis), the result is a sine wave. Rotation and oscillation are the same thing viewed from different perspectives.

y = amplitude × sin(angle)
A rotating point projected onto one axis produces a sine wave — the foundation of oscillation.

This means everything we build next—pendulums, springs, waves, harmonic motion—is built directly on what we learned today about angles, angular velocity, and angular acceleration. The angular motion pipeline is the engine; oscillation is what it produces.

The Bridge

Rotation → Projection → Sine wave → Oscillation. That's the chain. Today we learned rotation. Next, we project it, and everything clicks into place.

Now that we have the rotation foundation, let's put it to use. The next topic—springs—will show how a simple restoring force produces the exact same sine-wave oscillation we just visualized, no trig function required.

SECTION 09

What Is a Spring?

When you apply a force to an object, it can stretch, compress, or bend. This is called deformation. There are two kinds:

  • Elastic deformation: the object returns to its original shape once the force is removed—like a rubber band snapping back or a spring bouncing.
  • Inelastic (plastic) deformation: the object stays permanently changed—like a bent paper clip or squished clay.

A spring has a natural length (its rest state, with no force applied). When you pull it, the increase in length is called the extension. When you push it shorter, that's compression. In both cases, the spring fights back with a restoring force—it always wants to return to its natural length.

Two Forces Required

To deform an object without it flying away, you always need at least two forces. Stretching: two forces pulling in opposite directions. Compressing: two forces pushing inward. Even pushing something against a surface counts—the surface pushes back (Newton's third law).

In our p5.js simulations, we'll model ideal elastic springs: they always obey Hooke's Law, never break, and never permanently deform. It's a simplification, but it produces beautiful oscillating motion.

SECTION 10

Hooke's Law: F = −kx

Robert Hooke figured this out in 1660: "As the extension, so the force." The force of a spring is directly proportional to how far it's been stretched or compressed from its rest length. Double the extension, double the force.

Fspring = −k × x
k = spring constant (stiffness), x = extension (current length − rest length)

The Spring Constant (k)

The spring constant tells you how many Newtons of force it takes to stretch the spring by 1 metre. A low k (like 5 N/m) means the spring is soft and stretchy—think of a slinky. A high k (like 500 N/m) means the spring is stiff—think of a car suspension coil. In our code, k is just a tuning knob: higher values = snappier, more rigid springs.

The Negative Sign

The −1 is the most important part. It makes the force a restoring force: if the bob is below the rest position (positive extension), the force pulls it up. If the bob is above (negative extension / compression), the force pushes it down. Always back toward equilibrium.

let force = p5.Vector.sub(bob.position, this.anchor);
let currentLength = force.mag();
let x = currentLength - this.restLength;

force.setMag(-1 * this.k * x);
bob.applyForce(force);
The Elastic Limit

In the real world, Hooke's Law only applies up to the elastic limit. Stretch a spring too far and the force-extension relationship stops being linear—the spring deforms permanently. On a force-extension graph, this is where the straight line starts to curve. In our simulations, we ignore this, but it's worth knowing why real springs can break.

SECTION 11

Building a Spring Simulation

To implement a spring in code, we separate it into two objects: a Spring (knows the anchor position, rest length, and stiffness) and a Bob (a Mover from Chapter 2 with position, velocity, and acceleration). The Spring calculates the force and applies it directly to the Bob.

class Spring {
  constructor(x, y, length) {
    this.anchor = createVector(x, y);
    this.restLength = length;
    this.k = 0.2;
  }

  connect(bob) {
    let force = p5.Vector.sub(bob.position, this.anchor);
    let stretch = force.mag() - this.restLength;
    force.setMag(-1 * this.k * stretch);
    bob.applyForce(force);  // apply internally!
  }
}

The main loop is clean—same pipeline as Chapter 2:

function draw() {
  let gravity = createVector(0, 1);
  bob.applyForce(gravity);   // gravity always
  spring.connect(bob);       // spring force
  bob.update();
  bob.show();
  spring.show();
}
Same Pipeline, New Force

The Bob is just a Mover from Chapter 2. It still uses applyForce(), update(), and show(). The spring force is accumulated alongside gravity, friction, or any other force. This is the beauty of the force accumulation pattern: adding a new force doesn't require changing the object's code at all—you just call applyForce() one more time.

Notice that connect() calls applyForce() internally instead of returning the force vector. This is a design choice—you could do it either way. Embedding it inside the Spring keeps the draw loop cleaner, especially when you have multiple spring connections.

Damping in Springs

Without damping, a spring oscillates forever. In reality, air resistance and internal friction gradually steal energy. The quick fix: multiply the bob's velocity by 0.99 (or similar) each frame, just like angular damping. Or you can apply a proper friction force from Chapter 2. Either works—one is a shortcut, the other is physically grounded.

SECTION 12

Springs Produce Oscillation

Here's the payoff for the entire chapter: a bob on a spring naturally oscillates in a sine-wave pattern. No sin() function needed. Hooke's Law produces it automatically.

Why? Because the restoring force is proportional to displacement. The further the bob is from rest, the harder the spring pulls it back. It overshoots equilibrium (because of velocity), compresses past the rest point, and the spring pushes it back the other way. This back-and-forth motion is called simple harmonic motion—the same pattern described by the sine function.

x(t) = A × sin(ωt)
Amplitude × sin(angular frequency × time) — the position of an ideal spring bob over time.

If you track the bob's Y position over time and plot it, you get a perfect sine wave. This closes the loop on everything in this chapter:

  • Rotation around a circle → project onto one axis → sine wave
  • Spring with restoring force → bob oscillates → plots a sine wave
  • The sin() function → directly outputs a sine wave

Three completely different mechanisms, same curve. That's not a coincidence—it's the deep structure of oscillation.

The Full Circle

Radians → Rotation → Angular motion → Sine wave → Spring force → Oscillation. Every concept in this chapter feeds into the next. Radians let us describe rotation. Rotation projected onto an axis gives us sine. And Hooke's Law, a completely separate physical law, produces the same sine pattern through force and acceleration. Once you see it, you can't unsee it.

That's Chapter 3. The whole thing boils down to: radians are the natural language of rotation, angular motion mirrors linear motion with scalars, and springs are nature's built-in oscillators. The force-based approach from Chapter 2 carries forward: springs are just another force to accumulate.
