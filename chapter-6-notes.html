SECTION 01

Why Start with Neural Networks?

Up to now in the course, our systems have been mostly about motion: vectors, forces, rotation, steering. Neural networks shift the conversation from how something moves to how something decides.

That does not mean we are suddenly trying to reproduce the human brain with scientific accuracy. Just like Daniel Shiffman does in the Nature of Code chapter on neural networks, we use the brain only as inspiration. The goal is a simple computational model that can take numeric inputs and produce a useful decision.

The natural place to begin is the perceptron: the smallest possible neural network, a model of a single artificial neuron. It is simple enough to understand completely, yet rich enough to introduce the big ideas behind machine learning:

  • Inputs: numbers that describe the current example.
  • Weights: adjustable parameters that control how strongly each input matters.
  • Output: a guess or classification.
  • Learning: weight adjustment based on examples with known answers.
How To Mentally Frame It

Do not think of the perceptron as a tiny magic brain. Think of it as a weighted decision-maker. It takes in numbers, combines them, and chooses one of two sides. Learning means it changes the weights so future choices get better.

In this chapter, we stay deliberately narrow. We stop at the perceptron itself and the admissions example from class. No hand classifier, no large library, and no multi-layer networks yet. That narrow scope is a strength: once the one-neuron case is clear, the later extensions make much more sense.

SECTION 02

Inputs, Weights, and Bias

The admissions example from class has two real features:

  • Exam 1 score
  • Exam 2 score

Those become the perceptron's first two inputs. Then we add one extra input called the bias, which always has value 1. That means the full input list is:

inputs = [exam1, exam2, 1]
The third input is the bias. Its value is always 1, but its weight can change.

Each input has a matching weight. The weights tell the perceptron how much to care about each input. A larger positive weight means the input pushes the decision more strongly toward +1. A negative weight pushes the decision toward -1.

The bias is especially important. Without it, the perceptron's dividing boundary is too constrained. With it, the model can shift that boundary to a better place in the plane. You do not need slope algebra to understand this. Conceptually, the bias is what lets the perceptron move the split instead of always anchoring it in the same place.

Why Bias Matters

If the first two inputs are both zero, the bias still contributes because it is multiplied by its own weight. That means the perceptron can still make a nontrivial decision even at the origin. In practice, the bias is what gives the model flexibility.

So a perceptron with two features is really a three-input system: two measured values plus one bias input.

SECTION 03

Feedforward: Weight -> Sum -> Activate

The perceptron processes information in one direction only: the inputs go in, the output comes out. That is why it is called a feedforward model.

There are three conceptual steps:

  1. Weight each input: multiply every input by its matching weight.
  2. Sum the weighted inputs: add those products together.
  3. Activate: return +1 or -1 based on the sign of the sum.
sum = x0*w0 + x1*w1 + 1*wBias
All three contributions are added into one weighted total.

Then we apply a very simple activation function:

output = sign(sum)
If the sum is nonnegative, return +1. Otherwise return -1.
feedforward(inputs) {
  let sum = 0;
  for (let i = 0; i < this.weights.length; i++) {
    sum += inputs[i] * this.weights[i];
  }
  return this.activate(sum);
}

activate(sum) {
  return sum >= 0 ? 1 : -1;
}
Weighted Voting Machine

A good way to picture the perceptron is as a voting system. Each input casts a vote, but not all votes are equal. The weights determine how strong each vote is, and the final sum decides which side wins.

SECTION 04

Supervised Learning and Error

A perceptron does not magically know the correct rule when it starts. The weights are initialized randomly. To improve them, we use supervised learning.

Supervised learning means every training example comes with a known answer. In the admissions example, each student already has a label:

  • +1 for admitted
  • -1 for rejected

The perceptron makes a guess, then compares that guess to the known label. The difference between them is the error:

error = target - guess
If the guess is correct, the error is 0. If not, the error tells us the direction of correction.

Because the output is either +1 or -1, the error is easy to interpret:

  • If guess = target, then error = 0 and no change is needed.
  • If the perceptron predicts -1 but the answer is +1, then error = 2.
  • If the perceptron predicts +1 but the answer is -1, then error = -2.
Connection to Earlier Chapters

This should feel familiar. In the steering chapter, we used the difference between desired velocity and current velocity to compute a correction force. Here, the error plays a similar role: it tells the perceptron how to nudge its parameters in a better direction.

The crucial point is that the perceptron learns only because a teacher supplies the correct answers during training.

SECTION 05

Weight Updates and Learning Rate

Once we know the error, we adjust the weights. The perceptron learning rule is:

newWeight = weight + (error * input) * learningRate
The error, the input value, and the learning rate together decide how much each weight changes.

The learning rate is a small constant, often something like 0.01. It controls the size of the update:

  • Large learning rate: faster movement, but riskier and less stable.
  • Small learning rate: slower progress, but smoother and more controlled.
train(inputs, target) {
  let guess = this.feedforward(inputs);
  let error = target - guess;

  for (let i = 0; i < this.weights.length; i++) {
    this.weights[i] += error * inputs[i] * this.learningRate;
  }
}
Do Not Over-Mystify This

The perceptron is not "thinking" in a human sense. It is simply repeating a very mechanical process: guess, compare, update. The intelligence comes from doing that loop many times over data, not from any one dramatic step.

SECTION 06

The Admissions Example

The class exercise frames the perceptron as a university admissions classifier. Each student has two exam scores and a label indicating whether that student belongs to the admitted group or the rejected group.

The raw scores live in the familiar 0 to 100 range, but the code remaps them to the interval -1 to 1 before feeding them into the perceptron:

targetArray.push({
  exam1: row[0],
  exam2: row[1],
  x: map(row[0], 0, 100, -1, 1),
  y: map(row[1], 0, 100, -1, 1),
  label: label
});

This normalization step is not just cosmetic. It puts both inputs on a comparable scale, which makes learning more stable and the weight values easier to interpret.

Why Normalize?

If one input had a huge numeric range while another stayed tiny, the larger one would dominate the weighted sum much more easily. Mapping both features into the same range helps the perceptron treat them more fairly.

The Meaning of the Output

When the perceptron outputs +1, it is saying "this student belongs on the admit side of the boundary." When it outputs -1, it is saying "this student belongs on the reject side." The model is not producing nuanced explanations. It is drawing a binary split through the data.

That is why the visual demo is so helpful. The colored background shows the current classification regions, and the thin line marks the current decision boundary. You do not need to derive a slope formula to understand the main idea: the perceptron is trying to place one boundary that separates the two groups well.

SECTION 07

Training Set vs Test Set

One of the best parts of the class exercise is that it does not just show training. It also shows evaluation.

The data is split like this:

  • Training set: 32 students total, 16 admitted and 16 rejected.
  • Test set: 8 hidden students total, 4 admitted and 4 rejected.

The perceptron trains only on the training set. It repeatedly loops through those examples, updating its weights after each one. Then, once it reaches perfect accuracy on the training set in this controlled example, we stop and check the hidden test set.

Why Keep the Test Set Hidden?

If you train on the test set too, then it is no longer a test. The whole point of the hidden data is to ask: did the perceptron learn a rule that generalizes to new cases?

This is a small classroom dataset, so the split is not about industrial-scale statistical rigor. It is about learning the habit of separating learning data from evaluation data.

That habit matters. A model that gets everything right on the training set but fails on unseen data has not really learned the right pattern. It has just become too specific to what it already saw.

SECTION 08

The Perceptron Class

What makes the perceptron such a good teaching model is that the whole core implementation fits in a very small class. The class from the exercise stores weights, exposes a feedforward method, uses a sign activation, and updates weights in a training method.

class Perceptron {
  constructor(numInputs, learningRate) {
    this.weights = [];
    this.learningRate = learningRate;

    for (let i = 0; i < numInputs; i++) {
      this.weights[i] = random(-1, 1);
    }
  }

  feedforward(inputs) {
    let sum = 0;
    for (let i = 0; i < this.weights.length; i++) {
      sum += inputs[i] * this.weights[i];
    }
    return this.activate(sum);
  }

  activate(sum) {
    return sum >= 0 ? 1 : -1;
  }

  train(inputs, target) {
    let guess = this.feedforward(inputs);
    let error = target - guess;

    for (let i = 0; i < this.weights.length; i++) {
      this.weights[i] += error * inputs[i] * this.learningRate;
    }
  }
}

That is the whole learning system. There is no hidden magic outside the class. Every important idea in the chapter is visible here:

  • Random initialization of weights
  • Weighted sum of inputs
  • Sign activation for binary output
  • Error-driven updates during training
Why This Matters

You are not using a black-box library here. You can see the entire learning loop end to end. That is rare and valuable. Once you understand a one-neuron model at this level, larger neural networks stop feeling mysterious.

SECTION 09

What One Perceptron Can and Cannot Do

A single perceptron can learn only problems that can be separated by one straight decision boundary. The admissions example fits that idea well enough, which is why the demo works.

But not every classification problem is that simple. A famous counterexample is XOR. In XOR, the two classes are arranged so that no single straight line can separate them correctly. A one-neuron perceptron will fail there, not because it was trained badly, but because the model itself is too limited.

The Limitation Is Structural

If one straight boundary is not enough, no amount of extra training will fix the problem. The issue is not the data or the learning rate. The issue is that a single perceptron is not expressive enough for the task.

That limitation is not a weakness of this chapter. It is the reason the chapter is useful. The perceptron gives you the cleanest possible view of the fundamental ingredients of neural networks:

  • inputs
  • weights
  • bias
  • weighted sum
  • activation
  • supervised learning
  • train/test evaluation

Those ingredients do not disappear when networks get bigger. They get repeated, layered, and generalized. So the perceptron is not the end of neural networks, but it is exactly the right beginning.
